SYSTEM REQUIREMENTS DOCUMENT Project:  Front End Payment Management System for Village Health Workers (VHWs ) Client:  Ministry of Health and Child Care ( MoHCC ) Purpose:  To strengthen internal controls, eliminate payment loopholes, and prepare for  EcoCash  system integration while maintaining  decentralised  operations. 1. BACKGROUND AND CONTEXT The Ministry of Health and Child Care currently maintains a  large,  decentralised  Excel based database  of approximately  20,000 Village Health Workers (VHWs)  used to process  quarterly payments . While the database is regularly updated and serves as a reliable data source, the  existing manual process exposes the Ministry to significant operational and control risks . These risks include: Post verification alterations to payment lists Continued payment of exited or inactive beneficiaries Weak segregation of duties Heavy reliance on manual validation and trust based controls The Ministry therefore requires a  front end system  that sits on top of the existing database to enforce controls, manage workflows, and prepare for  secure system to system integration with  EcoCash . 2. OBJECTIVES OF THE SYSTEM The system shall: Address existing  control loopholes  in the payment process Enforce  segregation of duties  across Finance, HR, and Provinces Prevent payment of  ineligible beneficiaries Provide  audit ready records and reports Maintain  decentralised  provincial operations Prepare the Ministry for  EcoCash  API integration 3. SCOPE OF THE SYSTEM In Scope Front end web based application User management and role based access Payment workflow management Reporting and reconciliation Integration readiness (API ready architecture) Out of Scope (for Phase 1) Actual  EcoCash  API integration (to follow after system readiness) Replacement of the existing database (data will be reused) 4. USERS AND ROLES (ROLE BASED ACCESS CONTROL) 4.1 Provincial Officers Who is initiating the initial  payment What  do  they consider for the  vhw  to be paid. What is submitted as evidence View VHW records for their province Verify payment lists Submit lists for Head Office certification ❌  Cannot add or edit beneficiary details after submission 4.2 HR /  Programme  Custodians Maintain beneficiary master data Add new VHWs Update beneficiary status (Active / Inactive / Exited) ❌  Cannot initiate payments 4.3 Finance Officers Initiate payment batches Select province/district for payment View payment status and reports ❌  Cannot add or edit beneficiary records 4.4 National Oversight / Administrators Read only access to all data Unlock records after certification (with justification) View full audit logs 5. FUNCTIONAL REQUIREMENTS 5.1 User Authentication and Access Secure login using individual credentials Role based access enforcement Session timeouts and password policies 5.2 Beneficiary Management System shall display beneficiary details from the existing database Mandatory fields to include:  Full name National ID EcoCash  mobile number Province District Village they are overseeing Status (Active / Inactive / Exited) Only  Active  beneficiaries shall be eligible for payment 5.3 Payment Cycle Management System shall support  quarterly payment cycles Each cycle shall have:  Unique cycle ID Defined period (e.g. Q2 2026) A beneficiary may only be paid  once per cycle 5.4 Verification and Certification Workflow Provincial officers submit verified payment lists Head Office reviews and certifies lists Once certified:  The system  locks the records No changes are allowed for that cycle Any exception: Must be performed by a National Administrator Must include justification and be logged 5.5 Payment Initiation Finance officers can initiate payment batches:  By province By district National batch (if approved) System validates:  Beneficiary status Duplicate payments Mandatory data completeness 5.6 Reporting and Reconciliation System shall generate: List of paid beneficiaries Failed or excluded beneficiaries (with reasons) Payment summaries by:  Province District National total Reconciliation reports comparing:  Approved payment lists Executed payment batches Reports must be exportable (PDF/Excel). 5.7 Audit Trail The system shall log: User actions (add, edit, verify, certify, pay) Time and date of each action Old and new values for edits Reasons for overrides and unlocks Audit logs must be tamper proof. 6. NON FUNCTIONAL REQUIREMENTS 6.1 Security Role based access control Encrypted credentials Secure data storage 6.2 Availability and Reliability System accessible during business hours Graceful handling of incomplete sessions Regular data backups 6.3 Usability Simple, intuitive user interface Designed around  real use cases  (Finance, HR, Province) Avoid unnecessary complexity 7. INTEGRATION READINESS (FUTURE PHASE) System must be designed to:  Send payment instructions via API Receive payment status responses EcoCash  APIs will be integrated:  In a  UAT environment first After NDA signing After formal readiness confirmation 8. ASSUMPTIONS AND DEPENDENCIES Existing database remains the authoritative source Clean, approved data will be provided to ICT User requirements must be signed off before development EcoCash  integration depends on system readiness 9. SUCCESS CRITERIA The system shall be considered successful if it: Eliminates post verification data changes Prevents payment of exited beneficiaries Enforces segregation of duties Produces audit ready reports Is accepted and actively used by end users
