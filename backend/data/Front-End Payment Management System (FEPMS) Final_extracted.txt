Software Requirements Specification (SRS) Project:  Front End Payment Management System (FEPMS) Client:  Ministry of Health and Child Care ( MoHCC ) Purpose:  Strengthen internal controls, eliminate payment duplications, and prepare for secure EcoCash system integration while maintaining decentralised operations. 1. Introduction 1.1 Purpose The purpose of this document is to define the requirements for the  Front-End  Payment Management System (FEPMS) to support  MoHCC  in managing payments for Village Health Workers (VHWs). The system will enforce internal controls, prevent duplication, and prepare for integration with EcoCash. 1.2 Scope The FEPMS will be a web-based front-end application with role-based access control, payment workflow management, duplicate detection, reporting, and reconciliation. EcoCash API integration will be implemented in a future phase. 1.3 Definitions, Acronyms, Abbreviations MoHCC : Ministry of Health and Child Care VHW : Village Health Worker EcoCash : Mobile money platform operated by Econet FEPMS : Front End Payment Management System 1.4 References MoHCC  operational guidelines for VHW payments Econet EcoCash API documentation (future integration) 1.5 Overview The system will replace manual Excel-based processes with a controlled, auditable, and secure payment management system. 2. Overall Description 2.1 Product Perspective The FEPMS will operate as a front-end application connected to the existing database. It will enforce workflows and prepare payment instructions for EcoCash integration. 2.2 Product Functions Beneficiary management Payment cycle management Verification and certification workflow Payment initiation Reporting and reconciliation Audit trail 2.3 User Classes and Characteristics Provincial Officers : Initiate and verify payment lists. HR/Programme Custodians : Maintain master data. Finance Officers : Initiate payment batches and validate lists. National Administrators : Oversight, audit monitoring, and exception handling. 2.4 Operating Environment Web-based application accessible via standard browsers. Secure hosting environment with encrypted storage. 2.5 Design and Implementation Constraints Existing database remains authoritative. EcoCash integration dependent on readiness and agreements. 2.6 Assumptions and Dependencies Clean, approved data provided to ICT. User requirements signed off before development. 3. Specific Requirements 3.1 Functional Requirements 3.1.1 User Authentication and Access FR-01 : The system shall provide secure login with role-based access. FR-02 : The system shall enforce session timeouts and password policies. 3.1.2 Beneficiary Management FR-03 : The system shall display beneficiary details (Name, National ID, EcoCash number, Province, District, Village, Status). FR-04 : The system shall restrict payments to Active beneficiaries only. FR-05 : The system shall automatically flag duplicates based on National ID or EcoCash number. 3.1.3 Payment Cycle Management FR-06 : The system shall assign a unique cycle ID per quarter (e.g., Q2 2026). FR-07 : The system shall ensure a beneficiary can only be paid once per cycle. FR-08 : The system shall detect duplicates across provinces/districts. 3.1.4 Verification and Certification Workflow FR-09 : Provincial officers shall submit verified lists with supporting evidence. FR-10 : Head Office shall certify lists. FR-11 : The system shall lock records post-certification. FR-12 : Exceptions shall require administrator justification and audit logging. 3.1.5 Payment Initiation FR-13 : Finance officers shall initiate payment batches by province/district. FR-14 : The system shall validate beneficiary status, duplicates, and data completeness before payment initiation. FR-15 : The system shall prepare payment instructions for EcoCash integration. 3.1.6 Reporting and Reconciliation FR-16 : The system shall generate reports on paid, failed, and excluded beneficiaries. FR-17 : The system shall provide payment summaries by province, district, and national totals. FR-18 : The system shall reconcile certified lists with executed batches. FR-19 : Reports shall be exportable in PDF/Excel formats. 3.1.7 Audit Trail FR-20 : The system shall log all user actions (add, edit, verify, certify, pay). FR-21 : The system shall capture timestamps, old/new values, and override reasons. FR-22 : The system shall maintain tamper-proof audit records. 3.2 Non-Functional Requirements NFR-01 (Security) : The system shall encrypt credentials and enforce role-based access. NFR-02 (Reliability) : The system shall maintain 99.5% uptime during business hours. NFR-03 (Usability) : The system shall provide an intuitive interface for Finance, HR, and Provincial workflows. NFR-04 (Performance) : The system shall support 20,000 beneficiaries per cycle without degradation. NFR-05 (Backup/Recovery) : The system shall perform daily backups with recovery point objective (RPO) of 24 hours. 4. Future Integration Requirements FR-23 : The system shall be API-ready to send payment instructions to EcoCash. FR-24 : The system shall receive payment confirmations and error codes. FR-25 : Integration shall be tested in UAT before production rollout. 5. Success Criteria Duplicate payments eliminated across cycles. Payments prevented to exited beneficiaries. Segregation of duties enforced. Audit-ready reports produced. System accepted and actively used by  MoHCC  and Econet stakeholders.
